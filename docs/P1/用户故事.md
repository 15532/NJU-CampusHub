## 3.2 用户故事

以下用户故事基于“校园互助服务平台”需求规格说明书与需求调研方案整理，覆盖学生用户、需求方、接单方、平台管理员与自动化系统等主要角色，重点对应认证、任务流转、消息通知、信用评价、社区信息与后台管理等核心需求。

### US-01 统一身份认证
**作为** 注册学生，**我希望** 通过 CAS 或教育邮箱完成身份认证，**以便** 确保平台用户都是真实在校学生，提高互助环境的安全性。  
**优先级：Must**

**验收标准：**
1. 用户在注册时必须选择 CAS 认证或教育邮箱验证中的一种方式完成身份核验。
2. 认证成功后，系统为该用户创建实名账户，并允许其进入平台主页。
3. 认证失败时，系统应提示失败原因，并允许用户重新尝试。
4. 未完成认证的用户不能发布任务、接取任务或进入互动交易闭环。

### US-02 积分余额查询与变动记录
**作为** 学生用户，**我希望** 查看自己的积分余额和积分变动记录，**以便** 了解自己是否有足够积分发布需求，并追踪积分来源与去向。  
**优先级：Must**

**验收标准：**
1. 用户可以在个人中心查看当前积分余额。
2. 系统应展示最近的积分变动记录，包括时间、类型、变动数值和原因。
3. 当用户积分不足时，发布需要冻结积分的任务前应收到明确提示。
4. 积分记录中的每一条变动都应能对应到具体业务事件，如发布冻结、任务结算、违规扣分等。

### US-03 发布互助需求
**作为** 需求方，**我希望** 发布结构化的互助需求，**以便** 快速找到愿意帮助我的同学。  
**优先级：Must**

**验收标准：**
1. 用户可选择需求类别，如快递代取、学习辅导、二手交易、失物招领、组队等。
2. 发布表单至少包含标题、详情、地点/校区、时效、悬赏积分等必要字段。
3. 必填项未填写完整时，系统不得提交，并明确提示缺失项。
4. 提交成功后，需求应进入大厅展示，并显示为“待接单”状态。

### US-04 浏览与筛选需求
**作为** 学生用户，**我希望** 按校区、类别和关键字筛选需求，**以便** 更快找到与自己相关的任务或信息。  
**优先级：Must**

**验收标准：**
1. 需求大厅支持按校区、类别、关键字进行组合筛选。
2. 筛选结果应只展示符合条件的需求列表。
3. 当没有匹配结果时，系统应显示“暂无匹配内容”的提示。
4. 用户刷新页面后，系统应保留当前筛选条件或提供一键清空功能。

### US-05 接取互助任务
**作为** 接单方，**我希望** 接取符合条件的任务，**以便** 通过完成互助获取积分和信用提升。  
**优先级：Must**

**验收标准：**
1. 只有已实名认证且未被限制接单的用户才能点击“接单”。
2. 任务被成功接取后，其状态应从“待接单”变为“进行中”。
3. 同一时刻一个需求只能被一名接单方成功接取。
4. 接单成功后，需求方与接单方都应收到系统通知。

### US-06 发布时冻结积分
**作为** 需求方，**我希望** 在任务被接单后系统自动冻结悬赏积分，**以便** 保障接单者完成任务后可以正常结算。  
**优先级：Must**

**验收标准：**
1. 任务被接取时，系统应自动冻结需求方对应的悬赏积分。
2. 冻结后，用户可用积分应减少，但总积分记录中应保留冻结明细。
3. 若需求方积分不足，则该任务不能进入可接单状态。
4. 任务取消或结算完成后，冻结积分应被正确释放或转移。

### US-07 站内即时沟通
**作为** 需求方或接单方，**我希望** 在任务进行中进行站内沟通，**以便** 确认地点、时间和任务细节。  
**优先级：Should**

**验收标准：**
1. 任务双方在任务建立后可进入专属会话界面。
2. 系统至少支持文本消息发送与接收。
3. 新消息到达时，接收方应收到未读提醒或推送提示。
4. 未建立任务关系的普通用户不能直接进入该任务会话。

### US-08 提交完成凭证
**作为** 接单方，**我希望** 完成任务后上传照片等凭证，**以便** 证明任务已按要求完成并发起结算。  
**优先级：Must**

**验收标准：**
1. 对于跑腿、代取类任务，系统必须提供凭证上传入口。
2. 接单方至少上传 1 个有效凭证后，才能提交“完成任务”。
3. 提交后，任务状态应变为“待需求方确认”。
4. 上传失败时，系统应提示失败原因，且不得误将任务状态改为已完成。

### US-09 审核验收凭证
**作为** 需求方，**我希望** 审核接单方提交的完成凭证，**以便** 确认任务是否真实完成。  
**优先级：Must**

**验收标准：**
1. 需求方可查看接单方提交的全部凭证内容。
2. 需求方可以执行“确认完成”或“驳回申请”操作。
3. 确认完成后，系统应进入积分结算和互评流程。
4. 驳回时，需求方必须填写驳回原因，并通知接单方。

### US-10 超时自动结算
**作为** 接单方，**我希望** 在需求方长时间不处理时由系统自动结算，**以便** 避免我的劳动成果因拖延无法获得回报。  
**优先级：Should**

**验收标准：**
1. 接单方提交凭证后，系统应开始计算确认倒计时，如 24 小时。
2. 若倒计时结束前需求方未确认也未驳回，系统应自动判定任务完成。
3. 自动判定完成后，冻结积分应自动转入接单方账户。
4. 系统应向双方发送自动结算通知，并记录触发时间。

### US-11 双向评价
**作为** 需求方或接单方，**我希望** 在任务完成后对对方进行评分与评价，**以便** 帮助平台沉淀信用记录并辅助他人决策。  
**优先级：Must**

**验收标准：**
1. 任务结算完成后，双方都应收到评价提醒。
2. 评分范围应为 1 到 5 星，并支持填写文字评价。
3. 用户提交评价后，系统应更新被评价方的信用记录。
4. 同一用户对同一订单只能提交一次有效评价。

### US-12 违规举报与申诉
**作为** 学生用户，**我希望** 在遇到违规内容或纠纷时发起举报/申诉，**以便** 维护自身权益并改善平台秩序。  
**优先级：Should**

**验收标准：**
1. 用户可对订单纠纷、违规信息或异常行为发起举报或申诉。
2. 发起时需填写问题类型、说明文字，必要时可上传佐证材料。
3. 提交后系统应生成处理记录，并告知用户当前处理状态。
4. 相关处理结果应通过站内消息通知发起人。

### US-13 匿名发布失物招领/树洞信息
**作为** 学生用户，**我希望** 在特定板块匿名发布内容，**以便** 保护隐私同时继续获得帮助或表达需求。  
**优先级：Should**

**验收标准：**
1. 在失物招领、树洞或类似板块，系统应提供“匿名发布”选项。
2. 匿名发布后，前台只显示虚拟昵称或匿名标识，不显示真实身份信息。
3. 后台管理端仍可追溯发布者实名信息，用于风控与申诉处理。
4. 非匿名板块不得默认隐藏身份，避免误导交易对象。

### US-14 发布匿名树洞/失物信息
**作为** 学生用户，**我希望** 发布匿名树洞或失物招领信息，**以便** 在不暴露真实身份的情况下寻求帮助或分享信息。  
**优先级：Could**

**验收标准：**
1. 用户可在校园信息社区中选择失物招领或匿名树洞等内容类型。
2. 发布时系统应校验标题和正文不能为空。
3. 内容发布成功后，应在对应社区板块中展示。
4. 被判定为违规的内容应能被管理员下架。

### US-15 智能筛选与推荐
**作为** 学生用户，**我希望** 系统根据我的校区和历史偏好推荐任务，**以便** 更高效地发现我可能愿意接取或关注的内容。  
**优先级：Could**

**验收标准：**
1. 系统可根据用户常驻校区、历史浏览或接单偏好生成推荐列表。
2. 推荐列表中的内容应优先展示与用户相关度更高的任务。
3. 用户应能够切换到普通排序查看全部内容。
4. 当缺少历史行为数据时，系统应回退到通用推荐或最新发布排序。

### US-16 自动清理过期帖子
**作为** 平台管理员，**我希望** 系统自动清理或折叠过期帖子，**以便** 保持大厅信息的时效性和可读性。  
**优先级：Should**

**验收标准：**
1. 系统应依据任务截止时间或帖子有效期识别过期内容。
2. 过期后，内容应被自动折叠、隐藏或标记为“已过期”。
3. 已过期内容不得继续被正常接单。
4. 系统应保留过期处理日志，便于后台追踪。

### US-17 违规封禁处理
**作为** 平台管理员，**我希望** 对违规用户进行封禁或限制处理，**以便** 维护平台秩序与交易安全。  
**优先级：Must**

**验收标准：**
1. 管理员可查看用户违规记录、投诉记录和信用情况。
2. 管理员可执行警告、限制接单、临时封禁等处理动作。
3. 处理结果应即时生效，并通知被处理用户。
4. 所有处理操作都应记录操作人、时间和原因。

### US-18 数据看板与统计
**作为** 平台管理员，**我希望** 在后台查看订单量、活跃用户数和需求分类分布，**以便** 了解平台运营情况并调整管理策略。  
**优先级：Could**

**验收标准：**
1. 后台应展示基础统计指标，如日活、订单总数、分类分布。
2. 管理员可按时间范围查看统计数据。
3. 图表或表格数据应与实际业务数据保持一致。
4. 当无数据时，系统应显示空状态而不是报错。

### US-19 自动处理超时与过期业务
**作为** 自动化系统，**我希望** 定时检查超时任务和过期帖子，**以便** 减少人工运营成本并保证业务闭环。  
**优先级：Should**

**验收标准：**
1. 系统应按固定周期自动扫描待确认订单与即将过期的帖子。
2. 对超时未处理订单，系统应依据规则触发自动结算或状态流转。
3. 对超过有效期的帖子，系统应自动折叠、关闭或清理。
4. 自动处理动作应写入系统日志，供管理员审计。

## 说明：MoSCoW 优先级汇总

- **Must**：US-01、US-02、US-03、US-04、US-05、US-06、US-08、US-09、US-11、US-17
- **Should**：US-07、US-10、US-12、US-13、US-16、US-19
- **Could**：US-14、US-15、US-18
- **Won't**：本阶段暂无单独列出项
