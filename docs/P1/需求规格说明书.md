# 校园互助服务平台需求规格说明书

## 1. 引言

### 1.1 目的

本文档旨在明确“校园互助服务平台”的软件需求，详细描述系统的功能需求、非功能需求、外部接口及运行环境。本文档将作为系统设计、开发、测试以及验收的基础依据。

### 1.2 文档约定

- **优先级定义**：需求分为高（High）、中（Medium）、低（Low）三个优先级。
- **术语**：
  - **发布者 (Requester)**：在平台上发布互助需求的用户。
  - **接单者 (Provider)**：在平台上接受并完成互助任务的用户。

### 1.3 预期读者和阅读建议

本文档的主要读者包括项目经理、系统架构师、前端/后端开发人员、软件测试工程师以及 UI/UX 设计师。建议开发和测试团队重点阅读“第 3 部分：具体需求”。

### 1.4 产品范围

“校园互助服务平台”是一个专为在校大学生打造的综合性服务共享网络。平台旨在通过信息撮合，解决学生在校园生活和学习中的即时需求。核心业务涵盖：

- **快递代取**：解决最后一公里配送痛点。
- **失物招领**：快速找回校园内遗失物品。
- **学习辅导与问答**：促进跨专业、跨年级的知识共享与答疑。
- **二手交易**：推动校园闲置物品的循环利用。
- **组队匹配**：为竞赛（如大创、数学建模）、项目作业或体育活动提供组队渠道。

### 1.5 参考文献

- IEEE Std 830-1998: IEEE Recommended Practice for Software Requirements Specifications.

## 2. 总体描述

### 2.1 产品前景

本系统计划作为一个移动端优先的应用（微信小程序或独立 APP）进行开发，并配套一个 Web 端的管理员后台。系统将打破校园内的信息壁垒，建立基于校园信用体系的安全互助环境。

### 2.2 产品功能

系统主要包含以下宏观功能模块：

- **用户管理**：学号认证、个人资料维护、信用分管理。
- **需求大厅**：信息流展示、按校区分类检索、智能推荐、关键字搜索。
- **任务流转**：需求发布、接单、凭证上传、状态跟踪、取消机制。
- **交易与结算**：基于校园积分的流转与结算（替代真实的资金托管）。
- **即时通讯**：发布者与接单者的端到端在线聊天。
- **评价反馈**：双向评价机制、社区大众评审与申诉。

### 2.3 用户类和特征

- **学生用户（主体）**：具有实名认证身份的高校在读学生。对移动端操作熟练，对隐私保护和交易安全有较高要求。
- **志愿管理员/社区评审**：由高信用的学生兼任，负责处理一般纠纷和维护社区氛围。

### 2.4 运行环境

- **客户端**：支持主流移动操作系统的微信/支付宝小程序环境，或 iOS 13+/Android 10+ 原生应用。
- **服务端**：部署于 Linux 环境（如 Ubuntu），支持云原生架构。

### 2.5 设计和实现约束

- **合规性**：必须严格遵守《个人信息保护法》，用户敏感信息（如真实姓名）必须加密存储并严格限制访问权限。
- **封闭性**：系统仅限本校或特定高校联盟内的实名学生注册使用。

## 3. 具体需求

### 3.1 外部接口需求

#### 3.1.1 用户接口 (UI)

- 界面需符合现代移动端应用的交互规范，采用底部导航栏设计。 <small>[AI生成]</small>
- 各类需求的发布入口需明显，提供标准化的表单模板以降低用户的填写成本。 <small>[AI生成]</small>

#### 3.1.2 软件接口

- **统一身份认证接口**：需对接学校的教务系统（CAS/OAuth2）或发送教育邮箱（.edu.cn）验证码以验证学生身份。 <small>[AI生成+人工修改：删去人工照片审核，明确为CAS或教育邮箱验证]</small>
- **系统激励接口**：取消原定的第三方支付接口托管，改用平台内置的校园积分系统，支持积分流转或引导双方线下当面结算。 <small>[AI生成+人工修改：移除微信/支付宝支付，改为校园积分机制]</small>
- **即时通讯接口**：集成第三方 IM SDK（如腾讯云 IM 或环信）实现聊天功能。 <small>[AI生成]</small>

### 3.2 功能需求

#### 3.2.1 认证与账户模块

- **REQ-1 (高)**：系统应允许用户注册，并**强制要求通过 CAS 统一身份认证或教育邮箱验证**进行实名身份核验（废弃低效且易造假的照片人工审核方式）。 <small>[AI生成+人工修改：明确指定实名验证的技术方案]</small>
- **REQ-2 (高)**：系统应为每个用户维护一个信用分（初始为 100 分），履约加分，违约或被有效投诉扣分。低于阈值将限制接单权限。 <small>[AI生成]</small>
- **REQ-3 (高)**：系统必须支持“前台匿名、后台实名”的机制。在“树洞/情感倾诉/二手交易”等特定板块，用户可选择隐藏真实信息以虚拟昵称展示，保护隐私并提高表达意愿。 <small>[人工补充]</small>

#### 3.2.2 任务发布与大厅模块

- **REQ-4 (高)**：系统应提供结构化的需求发布表单，除了快递代取、学习辅导、二手交易、组队匹配外，**必须新增高频的失物招领、校园资讯/问答板块**。 <small>[AI生成+人工修改：增加了失物招领和问答两个高频校园场景]</small>
- **REQ-5 (高)**：需求大厅必须支持**按校区、宿舍楼栋进行 LBS 物理隔离筛选**。同时应具备对过期需求（如过时的拼单、已截止的组队）自动折叠或清理的机制，避免信息流污染。 <small>[AI生成+人工修改：增加了LBS校区筛选维度和过期数据清理逻辑]</small>

#### 3.2.3 任务流转与订单模块

- **REQ-6 (高)**：用户点击“接单”后，系统扣除并冻结发布者对应的悬赏积分，任务状态更新为“进行中”。 <small>[AI生成+人工修改：将冻结“悬赏资金”修改为冻结“悬赏积分”]</small>
- **REQ-7 (高)**：跑腿/代取类任务，接单者完成任务后必须上传完成凭证（如现场拍照图）。发布者确认后，积分转入接单者账户。 <small>[人工补充]</small>
- **REQ-8 (中)**：细化状态机流转：提交凭证 → 需求方确认/驳回 → **超时默认同意（如 24 小时未处理系统自动结算）** → 双方互评。若发生纠纷，废弃全职客服介入机制，**改由高信用用户组成的“社区大众评审”进行仲裁**，或系统根据信用分自动扣罚。 <small>[AI生成+人工修改：细化任务状态节点，将全职客服改为社区大众评审仲裁]</small>

#### 3.2.4 评价与互动模块

- **REQ-9 (高)**：订单完成后，双方需进行星级评分（1-5 星）和文字评价。评价直接影响用户的信用分，以此作为筛选靠谱用户的核心机制。 <small>[AI生成]</small>
- **REQ-10 (中)**：系统需提供内置聊天功能，支持发送文本、图片和语音消息，方便双方沟通细节。 <small>[AI生成]</small>

## 4. 非功能需求

### 4.1 性能需求 (Performance)

- **4.1.1 响应时间**：核心页面（如需求大厅刷新、发布需求详情页）的加载时间必须小于 2 秒。 <small>[AI生成+人工修改：将原1.5秒放宽至大作业指导书中要求的2秒标准]</small>
- **4.1.2 并发处理**：系统应能支持期末复习季、开学季等高峰期的高并发访问（目标支持至少 1000 QPS）。 <small>[AI生成]</small>
- **4.1.3 数据清理**：系统必须具备定时任务，自动折叠或清理超过时效性的冗余数据，保证检索性能。 <small>[人工补充]</small>

### 4.2 安全与隐私需求 (Security & Privacy)

- **4.2.1 数据加密**：用户密码禁止明文存储，需使用加盐哈希算法处理。防止 SQL 注入和 XSS 跨站脚本攻击。 <small>[AI生成]</small>
- **4.2.2 复合隐私机制（关键）**：对应 REQ-3，后台数据库必须保留实名追溯能力，以防违规行为，但前端展示层需严格执行匿名脱敏规则。 <small>[人工补充]</small>
- **4.2.3 接口鉴权**：所有 API 请求必须通过 JWT（JSON Web Token）进行鉴权，防止越权访问。 <small>[AI生成]</small>

### 4.3 可用性需求 (Usability)

- **4.3.1 极简操作**：移动端界面设计需简洁，用户的核心操作路径（如发布一条快递代取需求）从点击发布到完成，不超过 3 步。 <small>[人工补充]</small>
- **4.3.2 凭证上传便利性**：在任务流转中，提供快捷的唤起相机/相册功能，方便接单者一键上传完成凭证照片。 <small>[人工补充]</small>

### 4.4 可维护性需求 (Maintainability)

- **4.4.1 架构规范**：后端架构需遵循高内聚、低耦合的原则，前后端分离，关键业务模块必须编写单元测试。 <small>[AI生成+人工修改：明确补充了前后端分离架构和单元测试要求]</small>
- **4.4.2 接口文档**：必须提供自动化的接口文档（如 Swagger/OpenAPI），方便前后端协同开发与联调。 <small>[AI生成]</small>
