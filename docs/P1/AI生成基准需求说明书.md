# 校园互助服务平台需求规格说明书 (SRS)

## 1. 引言

### 1.1 目的
本文档旨在明确“校园互助服务平台”的软件需求，详细描述系统的功能需求、非功能需求、外部接口及运行环境。本文档将作为系统设计、开发、测试以及验收的基础依据。

### 1.2 文档约定
- **优先级定义**：需求分为高（High）、中（Medium）、低（Low）三个优先级。
- **术语**：
    - **发布者 (Requester)**：在平台上发布互助需求的用户。
    - **接单者 (Provider)**：在平台上接受并完成互助任务的用户。

### 1.3 预期读者和阅读建议
本文档的主要读者包括项目经理、系统架构师、前端/后端开发人员、软件测试工程师以及UI/UX设计师。建议开发和测试团队重点阅读“第3部分：具体需求”。

### 1.4 产品范围
“校园互助服务平台”是一个专为在校大学生打造的综合性服务共享网络。平台旨在通过信息撮合，解决学生在校园生活和学习中的即时需求。核心业务涵盖：
- **快递代取**：解决最后一公里配送痛点。
- **学习辅导**：促进跨专业、跨年级的知识共享与答疑。
- **二手交易**：推动校园闲置物品的循环利用。
- **组队匹配**：为竞赛（如大创、数学建模）、项目作业或体育活动提供组队渠道。

### 1.5 参考文献
- IEEE Std 830-1998: IEEE Recommended Practice for Software Requirements Specifications.

## 2. 总体描述

### 2.1 产品前景
本系统计划作为一个移动端优先的应用（微信小程序或独立APP）进行开发，并配套一个Web端的管理员后台。系统将打破校园内的信息壁垒，建立基于校园信用体系的安全互助环境。

### 2.2 产品功能
系统主要包含以下宏观功能模块：
- **用户管理**：学号认证、个人资料维护、信用分管理。
- **需求大厅**：信息流展示、分类检索、智能推荐、关键字搜索。
- **任务流转**：需求发布、接单、状态跟踪（进行中、待确认、已完成）、取消机制。
- **交易与结算**：虚拟货币或真实资金的托管与结算。
- **即时通讯**：发布者与接单者的端到端在线聊天。
- **评价反馈**：双向评价机制、举报与申诉。

### 2.3 用户类和特征
- **学生用户（主体）**：具有实名认证身份的高校在读学生。对移动端操作熟练，对隐私保护和交易安全有较高要求。
- **系统管理员**：负责平台日常运营，处理用户申诉，审核违规内容，管理系统参数。

### 2.4 运行环境
- **客户端**：支持主流移动操作系统的微信/支付宝小程序环境，或 iOS 13+/Android 10+ 原生应用。
- **服务端**：部署于 Linux 环境（如 Ubuntu），支持云原生架构。
- **数据库**：支持关系型数据库（如 MySQL/PostgreSQL）与缓存数据库（如 Redis）。

### 2.5 设计和实现约束
- **合规性**：必须严格遵守《个人信息保护法》，用户敏感信息（如学号、真实姓名）必须加密存储并严格限制访问权限。
- **封闭性**：系统仅限本校或特定高校联盟内的实名学生注册使用。

## 3. 具体需求

### 3.1 外部接口需求

#### 3.1.1 用户接口 (UI)
- 界面需符合现代移动端应用的交互规范，采用底部导航栏设计。
- 各类需求的发布入口需明显，提供标准化的表单模板以降低用户的填写成本。

#### 3.1.2 软件接口
- **统一身份认证接口**：需对接学校的教务系统或统一身份认证平台（CAS/OAuth2）以验证学生身份。
- **支付接口**：集成微信支付或支付宝第三方支付API，用于悬赏金的冻结与结算。
- **即时通讯接口**：集成第三方IM SDK（如腾讯云IM或环信）实现聊天功能。

### 3.2 功能需求

#### 3.2.1 认证与账户模块
- **REQ-1 (高)**：系统应允许用户通过手机号注册，并强制要求提交学号和校园卡照片（或对接校园认证接口）进行实名认证。
- **REQ-2 (高)**：系统应为每个用户维护一个信用分（初始为100分），履约加分，违约或被有效投诉扣分。低于阈值（如60分）将限制接单权限。

#### 3.2.2 任务发布与大厅模块
- **REQ-3 (高)**：系统应提供结构化的需求发布表单，根据分类动态调整字段：
    - **快递代取**：需包含取件点、送达点、物件大小、截止时间、悬赏金额。
    - **学习辅导**：需包含科目名称、辅导方式（线上/线下）、期望时间、薪酬预算。
    - **二手交易**：需包含物品图片、新旧程度、原价、现价、交易方式。
    - **组队匹配**：需包含项目/比赛名称、招募角色（如开发、文案、美工）、项目截止日期。
- **REQ-4 (中)**：需求大厅应支持按类别过滤、按时间/悬赏金额排序。

#### 3.2.3 任务流转与订单模块
- **REQ-5 (高)**：用户点击“接单”后，系统需通过支付接口冻结发布者的悬赏金额。任务状态更新为“进行中”。
- **REQ-6 (高)**：接单者完成任务后提交“完成申请”，发布者确认后，资金解冻并转入接单者账户。任务状态更新为“已完成”。
- **REQ-7 (中)**：若超时未确认，系统应在设定时间（如24小时）后自动确认完成。若发生纠纷，双方均可点击“申请客服介入”，冻结订单状态。

#### 3.2.4 评价与互动模块
- **REQ-8 (高)**：订单完成后，双方需进行星级评分（1-5星）和文字评价。评价直接影响用户的信用分。
- **REQ-9 (中)**：系统需提供内置聊天功能，支持发送文本、图片和语音消息，方便双方沟通细节。

### 3.3 性能需求
- **响应时间**：核心接口（如刷新大厅、发布需求）的响应时间不超过 1.5 秒。
- **并发处理**：系统应能支持期末复习季、开学季等高峰期的高并发访问（例如支持 1000 QPS）。
- **容量**：系统设计需满足单校区至少 20,000 名活跃用户的图文数据存储需求，图片需上传至云端对象存储（OSS）。

### 3.4 逻辑数据库需求
- 核心数据实体包括：用户（User）、需求（Task）、订单（Order）、消息（Message）、评价（Review）。
- 要求保持数据的一致性和完整性。订单表必须包含时间戳和明确的状态机流转记录。

### 3.5 软件系统属性

#### 3.5.1 安全性 (Security)
- 用户密码禁止明文存储，需使用加盐哈希（如 BCrypt）处理。
- 所有API请求必须通过JWT（JSON Web Token）或Session进行鉴权防越权访问。
- 防止SQL注入和XSS跨站脚本攻击。

#### 3.5.2 可靠性与可用性 (Reliability & Availability)
- 系统应保证 99.9% 的正常运行时间（Uptime）。
- 服务端应具有异常监控和日志收集机制，崩溃后应能实现快速自动重启。

#### 3.5.3 可维护性 (Maintainability)
- 后端架构需遵循高内聚、低耦合的原则，采用 RESTful API 设计规范，并提供自动化的接口文档（如 Swagger/OpenAPI）。
